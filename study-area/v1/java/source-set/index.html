[Gradle 筆記] 從實驗代碼到多模組架構：Java 專案目錄結構的最佳實踐

在 Java 專案開發中，標準的 src/main/java 結構有時無法滿足所有場景。當我們需要加入「實驗性質的代碼」、「教學課程」、「前端資源」，甚至是「將系統拆分為不同角色」時，該如何調整 Gradle 與目錄結構？

這篇文章整理了從 SourceSet 隔離 到 Multi-Module 多模組架構 的演進路徑與配置指南。

場景一：在同專案中隔離「練習」或「實驗」代碼

需求： 我想在現有的 Git Repository 中新增一個 src/learning/java 目錄，用來放練習題或實驗功能。我不希望這些代碼污染正式的 src/main/java，但它們需要能引用主程式的類別（如 Utils）。

解決方案：使用 Gradle SourceSets Gradle 允許自定義 SourceSet，將其視為獨立的編譯單元。

目錄結構：

src/
├── main/java/       (正式代碼)
└── learning/java/   (練習代碼)

build.gradle 配置：

sourceSets {
    learning {
        java { srcDir 'src/learning/java' }
        resources { srcDir 'src/learning/resources' }

        // 關鍵：讓 learning 可以讀取 main 的編譯結果與依賴
        compileClasspath += sourceSets.main.output
        runtimeClasspath += sourceSets.main.output
    }
}

// 繼承 main 的 dependencies
configurations {
    learningImplementation.extendsFrom implementation
}

場景二：像執行腳本一樣運行「特定課程」

需求： 在 src/learning 下有許多獨立的課程（例如 Lesson1, Lesson2），我希望透過指令 gradle runLesson1 直接執行，而不是每次修改 Main Class 設定。

解決方案：動態生成 Task 利用 Groovy 的靈活性，配合 Map 或檔案掃描動態建立 Gradle Task。

build.gradle 配置：

// 定義課程名稱與 Entry Point
def courses = [
    'Lesson1': 'com.learning.basic.HelloWorld',
    'Lesson2': 'com.learning.algo.DfsDemo'
]

courses.each { lessonName, mainClassPath ->
    tasks.create(name: "run${lessonName}", type: JavaExec) {
        group = 'Learning Courses'
        classpath = sourceSets.learning.runtimeClasspath
        mainClass = mainClassPath
    }
}

使用方式：

./gradlew runLesson1

場景三：處理第三層目錄 (如 src/main/web)

需求： 標準結構是 src/main/java 和 src/main/resources。如果我有 src/main/web 或 src/main/frontend，該如何處理？

觀念釐清： Gradle 第三層目錄通常代表 語言 (Language) 或 用途 (Function)。

三種常見做法：

目錄範例 應用場景 配置策略
src/main/web 靜態資源


(HTML/CSS/JS) | 加入 Resources：


sourceSets.main.resources.srcDir 'src/main/web'


編譯時會被複製到 Classpath。 | | src/main/webapp | 傳統 Web App


(JSP/WAR) | 使用 War Plugin：


plugins { id 'war' }


這是 Maven/Gradle 的標準目錄，自動識別。 | | src/main/frontend | 現代前端


(React/Vue/Angular) | 使用 Node Plugin：


不視為 Java 資源，而是透過 gradle-node-plugin 觸發 npm build，再將產物複製進 jar。 |

場景四：系統角色拆分 (API, Consumer, WebUI)

需求： 我的系統包含 api (對外服務), consumer (背景處理), web-ui (前台)。它們合起來是一個完整系統，但部署時是獨立的 Artifacts (Jar/Image)。

錯誤做法： ❌ 將它們放在 src/main/api, src/main/consumer。

  • 後果：依賴地獄 (Dependency Hell)，無法獨立打包，架構邊界模糊。

最佳實踐：Gradle Multi-Project Builds (多模組架構) 採用 Monorepo 策略，將每個角色提升為獨立的 Module。

推薦結構：

MySystem/
├── settings.gradle       (定義模組)
├── build.gradle          (共用配置)
├── common/               (共用邏輯, Utils)
├── api/                  (依賴 common + Web starter)
├── consumer/             (依賴 common + Kafka client)
└── web-ui/               (前端專案)

settings.gradle 配置：

rootProject.name = 'my-system'
include 'common', 'api', 'consumer', 'web-ui'

子模組依賴 (例如 api/build.gradle)：

dependencies {
    implementation project(':common') // 引用兄弟模組
    implementation 'org.springframework.boot:spring-boot-starter-web'
}

總結對照表

在決定目錄結構時，請依據「隔離程度」與「部署需求」來選擇：

架構模式 適用情境 優點 缺點
Standard


(src/main/java) | 一般應用程式 | 簡單，符合慣例。 | 所有代碼混雜，難以區分邊界。 | | SourceSets


(src/learning/java) | 實驗代碼、測試輔助、整合測試 | 在同一個 Jar 產出前進行邏輯隔離，不污染 Main。 | 仍然共用同一個 build.gradle，依賴管理較難細分。 | | Multi-Module


(:api, :common) | 完整系統、微服務、多種部署目標 | 實體隔離。每個模組有獨立的 Jar、依賴與 Task。 | 配置稍微複雜 (需維護 settings.gradle)。 |
