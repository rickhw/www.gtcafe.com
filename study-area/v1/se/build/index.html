Build

目的

  • 每次建制的 artifact image，都會自動簽入必要的 metadata
  • 依據 BuildType 提供不同檔案名稱的命名規格

用途

Artifact Metadata

以下為 Artifact Metadata 的資料結構

{
    // ## Product Metadata
    "app": {
        "productKeyName": "rws",
        "serviceKeyName": "booter"
    },
    // ## Build Metadata
    "build": {
        "version": "1.0.1",
        "buildType": "@BUILD_TYPE@",
        "buildId": "@BUILD_ID@",
        "hashcode": "@HASH_CODE@"
    }
}

這個檔案會放在 application 的肚子裡，在 build time 的時候，動態簽入 (sign) 相關資訊, 也就是置換 10-12 行的內容。

建置流程

用一個 build.sh 當作進入點，使用上有兩個例子：

Artifact Images - Naming

完成 build 之後的 artifact 檔案命名規則如下, 以 java 為例:

ls build/

## development build
rws-booter_v0.2.0-dev_b20240420-1200.jar
rws-booter_v0.2.0-dev_b20240420-1234.jar
rws-booter_v0.2.0-dev_b20240420-1521.jar
rws-booter_v0.2.0-dev_b20240421-0932.jar

## release build
rws-booter_v0.2.0-rel.jar

這個檔案名稱的規則如下:

  • Pattern: 依據 BuildType 使用不同的檔案名稱
    • dev build: {ProductName}-{ServiceName}_v{Version}-{BuildTyope}_b{BuildId}.{ExtName}
    • rel build: {ProductName}-{ServiceName}_v{Version}-{BuildTyope}.{ExtName}
  • ProductName: 產品名稱, 這是個 Key, 也就是唯一的代號
  • ServiceName: 產品底下的子元件 (Components) / 角色 (Roles) / 服務 (Service)
  • Version: 版本, Semantic Version
  • BuildType: 分成 dev, rel 兩種
  • BuildId: 建置當下的時間 (UTC)
    • 固定格式為 yyyyMMdd-HHmm, 可以用 $(date +%Y%m%d-%H%M) 取得
