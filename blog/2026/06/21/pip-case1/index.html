PIP - Case 1

June 21, 2026 · 7 min read
Rick Hwang
GTCafe Studio

記錄一下過去的用人踩雷的經驗，現在回想起來整個過程其實滿有趣的，先記錄第一個在前三個月就打算請他走人，姑且叫他 Lion。

註：故事大概是十幾年前的事，所以可能會有記憶失誤的問題，僅為個人的回憶與紀錄。然後十幾年前沒有 AI，不需要用有 AI 的現在來討論。

名校？​

Lion 這個人是國立大學資工所畢業的，當時他的畢業論文做的是跟 Raspberry Pi Cluster 分散式運算相關的題目，然後他有大約一年多的工作經驗，加上那時候團隊很需要人手，所以我錄用他了。當時還在團隊建立的很早期，所以「面試」流程並沒有很嚴謹，就是我自己一個人 OK 就二面了。這個人個性比較外向，跟一般工程師比較的話，比較會說話，然後很快我老闆二面後也就同意了。（至於哪一間國立大學先保留）

他入職大約兩個月後，同事們就經常向我反映他「不行、什麼都不懂」。一開始我還納悶：「怎麼會呢？好歹也是國立大學資工所畢業的 Master 啊！」直到有一次，我跟同仁正在看著螢幕討論事情，他突然湊過來，指著螢幕上的 Code 問我們：「哎，HTML 跟 HTTP 有什麼不一樣？」

當下，我跟那個同仁互看了一眼，心裡大概有一百萬匹草泥馬在奔馳。從那一刻起，我心裡就冒出一個念頭：「我要 fire 他！」

轉折：差點變社會新聞的 PIP 溝通​

有了汰換他的念頭後，加上不只一位同事反映他的狀況，我試著先讓他做一些小題目來驗證能力。結果證實，他真的不行。當時我跟老闆討論後，因為考量現場人力狀況，採取的策略是延長試用期，但是進入 PIP (績效改善計畫)。所以我就準備跟他溝通接下來的 PIP。

但在我正式想跟他談 PIP 之前，有天早上大約 8 點，辦公室都還沒什麼人的時候，他突然把我找去會議室，關上門，用一種快哭出來的狀態跟我說：他已經三天沒看到小孩了，因為他跟老婆吵架，老婆帶著小孩離家出走。

我心裡原本盤算著要跟他談 PIP，結果他先出了這招。我當下的直覺是：「PIP 的事我只能先吞下去。」如果不先把他的情緒安撫好，腦袋閃過一些劇本，最糟的就是萬一處理不當上了社會新聞怎麼辦？所以我花了一些時間安撫他，當時他的狀態確實非常糟糕。後來大約過了兩個禮拜，他老婆跟小孩都回家了，他的情緒總算比較平復了，我才正式開始跟他談 PIP 的事情。

意外的發展：從 RD 轉調業務​

PIP 的基本原則就是把規則講清楚：我會出一些基本的題目讓他驗證，如果沒過關，就得請他走人。結果在驗證過程中，他還是達不到標準。

正當準備請他離開時，他突然問我：「Rick，不然這樣好了，可不可以讓我轉去做業務？」

我聽了就說：「好吧，我幫你問問看。」我就去向老闆報告說這個人本來要被 PIP，但他自己提出想轉調業務的想法。老闆覺得只要同仁有意願，公司幫忙轉調沒問題。接著我老闆去問業務主管，業務主管也答應了，只不過業務靠的是業績獎金，起薪低，必須降薪。

於是，我們就終止了 PIP 流程，讓他直接從研發處轉調到業務處，這也算是一個不錯的收尾。

後來他在業務部待了一個月，跑來跟我說他第一個月的業績就達標了，我有去業務處看到他們掛在牆上的業績數字，他的名字和業績真的是有達標的，還滿厲害的。其實這個人的個性比較外放，某種程度上確實比較適合做業務。

有趣的是，到了他去業務處第二個月的某天早上，他突然跑來找我，在我桌上放了一瓶高粱酒，說要謝謝我教了他很多東西。他因為其他個人規劃要離職了，具體原因我忘了，但總之，整個事情算是告一個段落。回家後我看著那瓶高粱酒，心情其實有點複雜 XD

經驗總結：調整面試流程與實機測驗​

這次的經驗對我來說是個警惕：用人真的要很謹慎，總是會踩到一些雷。

頂大或名校畢業的學生，素質通常不會太差，但在面試過程中，還是必須要做實質的驗證。因為這次摔了個大跤，後來變成被同仁嘴的一種默契與笑話，但也因此後來我就找部門內部資深同仁，一起擬定了部門的第一版的面試流程，其中增加了「實機寫 Code」的關卡。名校畢業是個不錯的條件，但是驗證寫 Code 是過程必要的。

讓面試者現場寫 Code、實際敲鍵盤，我們可以觀察很多細節：

  • 寫 Code 的習慣好不好？
  • 有沒有熟悉的 Library？
  • 對於程式碼有沒有自我要求？
  • 遇到問題怎麼找資料？
  • 怎麼釐清需求？
  • 以及怎麼跟團隊合作？

我們不會出太難的題目，更不會考困難的 LeetCode 演算法，因為在我們一般的開發工作中，不太會用到那些東西。我們出題的難易度適中，而且「全程開放面試者 Google、查 Stack Overflow」。最重要的是，在測驗過程中，面試者隨時需要跟可以跟我們討論問題的定義，我們把「釐清需求」當作面試的一環。

後來部門的標準流程是：面試一開始，先由我的兩位資深同仁進行技術驗證，大約花 10 到 15 分鐘讓他寫 Code。如果這關 OK 了，我才會進去面談。

寧缺勿濫​

團隊從無到有的過程很辛苦，現場忙得要死，所以早期的確因為「需要人手」，用錯「幾次」。幾次的誤用，團隊困擾，我也困擾，老闆 / HR 也會覺得很煩。

但在團隊人力很少的時候，或者單看人數的時候，會有「先求有、再求好」的心態，但「用人」這件事，以這樣出發點想就會造成後續連鎖的問題。

幾次踩雷之後，我就恪守「寧缺勿濫」的原則，這也得力於當時我的左右手、和我老闆的幫忙，後來面試、用人的精準度都提升不少，更重要的是找到公司後來重要的人才。

番外篇：拒絕寫 Code 的奇葩面試者​

在這個新流程中，我們也遇過一些奇妙的 Case。

曾經有一位從台中搭車上來台北面試的求職者。一如往常，我的兩位資深同仁進會議室，把電腦交給他，請他當場寫考題。結果才進去不到 5 分鐘，兩位同仁就跑出來跟我說：「Rick，他拒絕寫這份考題，怎麼辦？」

流程走不下去了，同仁問我該怎麼辦。我心想，才 5 分鐘就結束也太尷尬了，為了避免公司名聲受損，於是我就進去收尾。

我們後來聊了大概半小時，當然最後是沒有錄取。但我花了一些時間跟他溝通面試的本質是什麼、為什麼我們要求實務測驗。我跟他說：「你都花錢、花時間從台中坐車上來了，而且我們在 JD（職缺描述）上也有寫明會有上機測試。你當場拒絕測驗，其實也是在浪費你自己的時間。」重點是不要讓他有白跑一趟的感覺。

我不知道他最後有沒有聽懂，但對我來說，我的策略是不能把公司的形象搞砸，所以我把他當作是一次幫部門和公司做公關形象的機會，好好地跟他說明我們的理念。

面試、用人還有其他有趣的例子，下次來記錄用到一個頂大資工所，最後 PIP 的例子。
